# Google Maps CN Satellite Corrector 谷歌地图卫星图纠偏

A Chrome / Edge (Manifest V3) extension that fixes the misalignment between
the **satellite imagery** and the **road/label layer** on
[google.com/maps](https://www.google.com/maps) inside mainland China.

修正谷歌地图在中国大陆地区卫星图层与道路/标注图层之间的位置偏移。

## Why the offset exists

Chinese regulations require map providers to publish street data in the
obfuscated **GCJ-02** ("Mars") coordinate system. On Google Maps the
road/label layer for the mainland is GCJ-02, but the satellite imagery is
true **WGS-84**, so the two layers disagree by roughly **100–700 m**
depending on location (e.g. ~540 m near Guangzhou).

## How this extension fixes it

Instead of moving the labels, the extension shifts the **satellite tiles**
to match the labels:

1. A script injected into the page's MAIN world intercepts satellite raster
   tile requests (`khms*.google.com/kh`, `mt*.google.com/vt?lyrs=s`, …) made
   via `fetch`, `XMLHttpRequest`, or `<img src>`.
2. For each requested tile it computes the local WGS-84 → GCJ-02 delta at
   the tile center, converts it to a pixel offset at that zoom level, and
   fetches the up-to-4 neighbouring source tiles that cover the shifted
   area.
3. It composites them on a canvas at sub-pixel precision and returns the
   rebuilt tile, so imagery lines up with roads and labels.

Tiles outside mainland China — and Taiwan / Hong Kong / Macau, where
Google's labels are *not* GCJ-02 shifted — pass through untouched. Corrected
tiles are cached in memory.

## Installation (load unpacked)

**Chrome:** open `chrome://extensions` → enable **Developer mode** →
**Load unpacked** → select this folder.

**Edge:** open `edge://extensions` → enable **Developer mode** →
**Load unpacked** → select this folder.

Then open the reference page and toggle the satellite layer:
<https://www.google.com/maps/@23.1232526,113.2375627,5600m/data=!3m1!1e3>
Roads and labels should now sit on top of the matching imagery. The console
logs `[GMaps CN Corrector] satellite offset correction active` when a tile
has been corrected.

Use the toolbar popup to enable/disable the correction (reload the Maps tab
after changing it).

## Development

No build step; plain JS. Run the tests with Node (≥ 18):

```sh
node test/core.test.js      # coordinate/tile math
node test/install.test.js   # fetch-hook integration against a stubbed scope
```

Icons are generated by `tools/make_icons.py` (requires Pillow).

## Files

| File | Purpose |
| --- | --- |
| `manifest.json` | MV3 manifest; injects the scripts at `document_start` |
| `corrector.js` | Pure GCJ-02/tile math + network hooks (MAIN world) |
| `bridge.js` | Isolated-world bridge mirroring the on/off setting into the page |
| `popup.html` / `popup.js` | Toolbar popup with the enable toggle |
| `test/` | Node test suites |

## Troubleshooting

**"We're sorry, but an error has occurred. Please reload the page."**
This was caused by v1.0.0's `Worker` constructor patch, which re-wrapped
Google's own worker scripts in `blob:` URLs and broke the Maps app. Fixed in
v1.0.1 (the patch was removed). Update to the latest version, go to
`chrome://extensions` / `edge://extensions`, click **Reload** (⟳) on the
extension, then reload the Maps tab.

If Maps still fails to load, disable the extension to confirm it is the
cause, then open an issue with the errors from the DevTools console (F12).

## Limitations

- The GCJ-02 transform is the publicly known approximation; residual error
  is typically a few meters (well under the original 100–700 m offset).
- 3D globe/tilt view uses a different tile pipeline and may not be fully
  corrected; the standard 2D satellite view is the supported mode.
- Google may change tile URL formats; `parseTileUrl()` in `corrector.js` is
  the single place to update.
